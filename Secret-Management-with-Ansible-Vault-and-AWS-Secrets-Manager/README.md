Secret Management with Ansible Vault and AWS Secrets Manager:

    Overview
    This project demonstrates two complementary secret management approaches used together: Ansible Vault for encrypting secrets at rest in version control, and AWS Secrets Manager for centralized cloud-based secret storage. A playbook retrieves the AWS secret at runtime and combines it with values from the Vault-encrypted file to configure an application.
    Key highlights:
    
    Ansible Vault creates AES256-encrypted YAML files — $ANSIBLE_VAULT;1.1;AES256 header confirms encryption
    AWS Secrets Manager stores secrets as JSON key-value pairs, retrieved via aws secretsmanager get-secret-value
    no_log: true on sensitive tasks prevents passwords from appearing in Ansible output or logs
    Vault password stored in ~/.vault_pass with chmod 600 — excluded from Git via .gitignore
    Config file deployed with mode: '0600' — only root can read it

Project Structure:

     Secret-Management-with-Ansible-Vault-and-AWS-Secrets-Manager/
    |
    |-- ansible.cfg              # Ansible config — vault_password_file, remote_user
    |-- retrieve_secrets.yml     # Playbook — fetch from Secrets Manager + deploy config
    |-- secrets_plain.yaml       # Plain text template — encrypt before use
    |-- .gitignore               # Excludes .vault_pass and plain secrets from Git
    |
    |-- README.md

    Note: secrets.yaml (the encrypted file) is generated by running ansible-vault encrypt on secrets_plain.yaml.
    It is safe to commit to version control. secrets_plain.yaml must never be committed.

Architecture

    ansible-control-node
            |
            |-- secrets.yaml (Ansible Vault AES256 encrypted)
            |   contains: db_username, db_password, api_key
            |
            |-- retrieve_secrets.yml
                  |
                  | aws secretsmanager get-secret-value
                  v
            AWS Secrets Manager (my-secret)
                  |
                  | {"username": "...", "password": "..."}
                  v
            set_fact: secrets = parsed JSON
                  |
                  |-- Create OS user with hashed password
                  |-- Deploy /etc/app/config.env (mode 0600)

Task 1 — Attach Secrets Manager Policy to IAM Role

    IAM → Roles → AnsibleControlRole → Add Permissions → Attach Policies
      Search: SecretsManagerReadWrite → Attach
    
    Verify:
      aws secretsmanager list-secrets --region your-region
      # Empty list is fine — no error means access is working

Task 2 — Set Up Project Directory
    
    mkdir -p ~/ansible-vault-lab
    cd ~/ansible-vault-lab
    mkdir -p {group_vars,host_vars,files}

Task 3 — Create and Encrypt the Vault File

    Create vault password file

    echo "your-vault-password" > ~/.vault_pass
    chmod 600 ~/.vault_pass
    # 600 = only owner can read and write — no other users
    Edit secrets_plain.yaml with your values
    Open secrets_plain.yaml and replace the placeholder values.

    Encrypt the file
    
    ansible-vault encrypt secrets_plain.yaml \
      --vault-password-file ~/.vault_pass \
      --output ~/ansible-vault-lab/secrets.yaml

    Verify encryption

    head -1 ~/ansible-vault-lab/secrets.yaml
    # Expected: $ANSIBLE_VAULT;1.1;AES256
    
    # View decrypted content
    ansible-vault view ~/ansible-vault-lab/secrets.yaml \
      --vault-password-file ~/.vault_pass

    Useful vault commands

    # Edit an encrypted file
    ansible-vault edit secrets.yaml --vault-password-file ~/.vault_pass
    
    # Change the vault password
    ansible-vault rekey secrets.yaml --vault-password-file ~/.vault_pass
    
    # Decrypt to plain text
    ansible-vault decrypt secrets.yaml --vault-password-file ~/.vault_pass

Task 4 — Store Secrets in AWS Secrets Manager
    
    # Primary application secret
    aws secretsmanager create-secret \
      --name "my-secret" \
      --description "Application credentials" \
      --secret-string '{"username":"your-app-user","password":"your-app-password"}' \
      --region your-region
    
    # API keys secret
    aws secretsmanager create-secret \
      --name "my-api-keys" \
      --secret-string '{"api_key":"your-api-key","app_secret":"your-app-secret"}' \
      --region your-region
    
    # Verify
    aws secretsmanager get-secret-value \
      --secret-id "my-secret" \
      --query SecretString \
      --output text \
      --region your-region

Task 5 — Install Session Manager Plugin

    curl "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/ubuntu_64bit/session-manager-plugin.deb" \
      -o "/tmp/session-manager-plugin.deb"
    
    sudo dpkg -i /tmp/session-manager-plugin.deb
    
    session-manager-plugin --version

Task 6 — Run the Playbook

    Update retrieve_secrets.yml — change your-region to your actual region.
    
    cd ~/ansible-vault-lab
    
    # Syntax check
    ansible-playbook retrieve_secrets.yml \
      --vault-password-file ~/.vault_pass \
      --syntax-check
    
    # Option A: Using vault password file
    ansible-playbook retrieve_secrets.yml \
      --vault-password-file ~/.vault_pass
    
    # Option B: Interactive prompt
    ansible-playbook retrieve_secrets.yml --ask-vault-pass
    
    # Option C: Environment variable
    export ANSIBLE_VAULT_PASSWORD_FILE=~/.vault_pass
    ansible-playbook retrieve_secrets.yml

Task 7 — Verify:

    # Confirm vault file is encrypted
    head -1 ~/ansible-vault-lab/secrets.yaml
    # $ANSIBLE_VAULT;1.1;AES256
    
    # Confirm secret exists in AWS
    aws secretsmanager get-secret-value \
      --secret-id "my-secret" --region your-region
    
    aws secretsmanager describe-secret \
      --secret-id "my-secret" --region your-region
    
    # Confirm user was created
    id your-app-user
    
    # Confirm config file was deployed
    sudo cat /etc/app/config.env
    # Should show -rw------- (mode 600) and be owned by root

Key Concepts:

    no_log: true
    Adding no_log: true to any task that handles passwords or keys prevents the values from appearing in Ansible output, log files, or the callback output.
    Always apply it to tasks that use password_hash, copy with secret content, or user with a password.
    
    Why AES256
    Ansible Vault uses AES256 symmetric encryption. The vault password is the key. 
    Without the password, the encrypted file reveals nothing about its contents — the header $ANSIBLE_VAULT;1.1;AES256 only identifies the format.

Cleanup:

    # Delete AWS secrets immediately (without 30-day recovery window)
    aws secretsmanager delete-secret \
      --secret-id "my-secret" \
      --force-delete-without-recovery \
      --region your-region
    
    aws secretsmanager delete-secret \
      --secret-id "my-api-keys" \
      --force-delete-without-recovery \
      --region your-region
    
    aws secretsmanager list-secrets --region your-region
    # Should return an empty list
    
    # Remove local files
    rm -rf ~/ansible-vault-lab
    rm -f ~/.vault_pass
    
    # Terminate EC2 instances via console
    # EC2 → Instances → ansible-control-node + web-server-01 → Term
